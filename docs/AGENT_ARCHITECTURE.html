What this system is

A conversational agent that classifies bank transactions through a self-improving rule engine and answers forward-looking financial questions grounded in 15 years of real transaction history. The AI is a reasoning interface to a structured knowledge base — not the product itself.

Technical architecture

All components run in Docker on the M720q home server. The web UI is served behind a Cloudflare tunnel. Real data stays on the host filesystem, never in the container image.

Functional architecture

The agent loop is the single execution path for both classification and scenario work. Tool groups expose different capabilities; the orchestrator decides which to call.

User journeys

Journey 1 — Classify unknown transactions (HITL loop)

The agent fetches unclassified rows, proposes a rule via Haiku, previews blast radius, presents to the user, and only writes on explicit approval. The code gate (B1) enforces this at the dispatch layer.

Journey 2 — Scenario: mortgage rate change

Multiple tools called in one response. The agent decides what data it needs, calls all tools, synthesises results, then writes discovered facts to agent_state for future sessions.

Journey 3 — Ingest real bank data

One-time setup or run after a new bank export. The CLI combines and classifies CSVs, runs the seed rules, and surfaces a Missing count for the agent to work through.

Design principles

Nine load-bearing decisions. Each has a concrete reason, an alternative that was rejected, and a pattern that generalises beyond this project.

1. Raw API tool use — no framework Building the agent loop manually made the mechanics concrete: build context → call API → execute tools if present → loop. ~40 lines of Python. Once you've written it, any framework (LangGraph etc.) is a known set of trade-offs. Frameworks were deferred to a future project where the fundamentals are already understood.
2. Hybrid session + cross-session memory Messages array handles within-session continuity (trivial, cheap). SQLite agent_state handles cross-session facts (explicit, queryable, correctable). Conflating them — treating long history as memory — is a common mistake. Each memory layer serves a different purpose and uses a different mechanism.
3. Preview-before-apply as the default for state mutation Every tool that writes to the database has a paired read-only preview. Initially added for classification rules only. Promoted to a project-wide default (B1 code gate) once the pattern proved its value. No apply_* runs without a preceding preview_* in the conversation history.
4. Phase 1 pipeline + rule lookup as a Strangler Fig The hardcoded if/elif chain became the fallback when SQLite rules were introduced, then was removed entirely in A1. New behaviour wrapped old behaviour. Nothing broke during the transition. This is the Strangler Fig pattern — the right way to modernise working code without a full rewrite.
5. Model routing by task complexity Claude Sonnet for the orchestration loop (complex multi-step reasoning). Claude Haiku for classification suggestions (short structured input/output — same quality, 20× cheaper). Prompt caching on system prompt and agent_state snapshot. The cost question is: what is the complexity ceiling of this specific task?
6. SQLite over alternatives (MongoDB, Postgres, JSON file) Transactions are naturally tabular. SQLite is single-file, zero-infrastructure, built into Python. MongoDB would add a server process for no benefit. PostgreSQL is multi-user infrastructure for a single-user tool. JSON file has no queryability. The right tool for relational data at personal scale.
7. Renderer protocol as the display abstraction Not in the original spec — emerged during Step 5. The agent loop calls renderer.show_response() without knowing whether it's writing to a terminal, an SSE stream, or a test fixture. This is why the web UI (C4), transcript replay (D2), and every subsequent renderer required zero changes to run_turn().
8. Demo mode at the data layer A data_source column on the transactions table. Synthetic data is committed to the repo. Real data arrives via Docker volume mount, gitignored. The demo is safe to share publicly without any auth system. Privacy is an architecture decision, not a security layer bolted on at deployment.
9. Audit trail separate from context All session events written to JSONL transcripts. Reasoning captured in structured log entries. None of this is fed back into the messages array — context stays clean and cheap. The audit trail answers "why did the agent do that?" without inflating token costs per turn.
